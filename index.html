The "science" of setting goals

Posted on 2024-02-13 In methodology

什么是目标？在我的知识体系中，目标是我期望的未来的状态。
为什么需要目标？没有目标的话，没有准确的期望的状态，也就没有方向，线索去指导我做出最佳的决策。当然人算不如天算，乱撞的人也有可能可以做出正确的决策，算尽一切的人也会因为算力以及决策方法和现实的复杂度而做出错误的决策。所以“尽人事听天命”是一个很好的态度。
目标(集)的(内部)结构以及关系？

  • 存在层次结构
    • 超出动物性的目标
    • 人生物本能的目标
    • 满足具体生物本能很多层次的细化目标
    • …
  • 存在关系
    • 子目标
    • 父目标
    • 兄弟目标
    • 无关目标
    • 依赖目标
      为什么目标制定需要自顶向下？
      如果指定目标的格局(层次较低)，容易造成 “有目标的错觉”。
      ”有目标的错觉“：在很多阶段，我们会有短期的目标，比如高中阶段是高考分数尽可能的高，大学尽可能保研到更好的学校(保研在我目前看来是十分短时的，没有格局的目标)。这些短期的目标往往是过去许许多多与我们处于同一阶段，相近客观情况，拥有相同更高层次的目标的人做出的明智选择。我们往往没有完整的推理链条从高格局(层次)的目标推理到低层次的目标，我们有的一般都是道听途说，非理性，非可重现的推理。这样的话，当环境发生剧烈的变化时，我们的目标往往会失效。如果不及时更正低层次目标，我们会向这错误的方向不断前进，消耗宝贵的时间，精力，资源。

（上面的总结并没有完全经过严谨的推理，不过是我总结的一些”经验“，不保证任何时候都是准确的。）

Paradigm Transitions

Posted on 2024-02-01 In methodology

[*******]，我浏览了很多的心理学理论。

其中有一个理论是 Dual Process Theory, 虽然很简单，却提供了一个有效的手段来区分人。如何区分人涉及到一个人对 逻辑系统的依赖程度，以及对 直觉系统的依赖程度。

这个理论的提出者是 Keith Stanovich, 他在 2009 年的一篇文章中提出了这个理论，他认为人的思考过程可以分为两个系统，一个是逻辑系统，一个是直觉系统。逻辑系统是一个慢速的系统，它需要大量的认知资源，而直觉系统是一个快速的系统，它不需要大量的认知资源。逻辑系统是一个有意识的系统，而直觉系统是一个无意识的系统。

我自我审视发现其实我并不是真正有个强大的逻辑系统，我只是有一个强大的直觉系统，我在很多时候都是通过直觉来做决策的，并且套上了逻辑系统的形式，看起来像是依赖于逻辑系统。而且这个现象是及其普遍的，很难避免的。我经常会使用“我觉得”，“我认为”，不摆数据，讲的话无法得到复现。我进而观察了其他的很多的人，我到现在为止并未发现有人有一套自洽的，真正形式化的逻辑系统。我现在正前进在构建一个自洽的逻辑系统的路上。

ATodo user report

Posted on 2023-12-31 In creation

已经使用了 ATodo 有一段时间了，感觉很不错。

优点：

  • 再也没有花时间想自己要怎么规划了，只需要按照 ATodo 的提示做就好了。
  • 任何想法都能第一时间整合进我的 ATodo 中，不会放弃一丝可能。
  • 多任务管理已经不存在，只要合理配置 ATodo 任务，任务数量根本不是限制。如果完不成，只能算是时间不不够或者是未检查到的计划冲突。
  • ATodo 使得长线规划成为可能，兼顾大局与细节。

问题：

  • 任务触发器种类以及自动触发不完善。
    • 国内的消息平台如微信和 QQ 过于封闭，没有提供消息接口，无法实现消息触发器。并且由于我以后也不会有机会使用这两个平台进行关键社交和工作，所以不会再考虑支持这两个平台。
    • 关于周期性任务的自动触发已有详尽，灵活的方案。但还有一些小功能没有实现。
    • 邮件触发器还未支持，并且以后可能会尝试推展 LLM(Large Language model) 的邮件触发器。
  • UI 过于丑，真直男审美。不过我本来也没做 UI 优化，等有精力了搞个现代化的 UI。
  • React 框架过于操蛋，我已经尽力在架构层面下功夫了，但是 React Hooks 业务与 UI 的耦合性太高，无法充分分离。相比于 Signal，思想负担太高，我的脑子有其他的事情要想，以后有机会会迁移到 Preact 上。
  • 持久化方案太烂，我现在是直接存储 json 对象，以后会转移到 Sqlite 上。

总的来说，ATodo 是一个很好的工具，不愧我投入的上十天的开发时间。

I’ve been using ATodo for a while now, and it feels great.

Pros:

  • I no longer have to spend time thinking about how to plan, I just need to do what ATodo suggests.
  • Any idea can be integrated into my ATodo immediately, not giving up any possibility.
  • Multitasking is no longer an issue, as long as ATodo tasks are reasonably configured, the number of tasks is not a limitation. If it can’t be completed, it’s only because of insufficient time or unchecked plan conflicts.
  • ATodo makes long-term planning possible, taking into account both the big picture and the details.

Problems:

  • The variety of task triggers and automatic triggering are imperfect.

    • Domestic messaging platforms like WeChat and QQ are too closed, without messaging interfaces, making it impossible to implement message triggers. And since I won’t have the chance to use these two platforms for critical social and work interactions in the future, I won’t consider supporting them anymore.
    • Regarding automatic triggering of periodic tasks, detailed and flexible solutions exist. But some small features are still missing.
    • Email triggers are not yet supported, and in the future, I might try to develop LLM (Large Language Model) based email triggers.

  • The UI is too ugly, truly a straight man’s aesthetic. But I never optimized the UI anyway, I’ll make a modern UI when I have the energy.

  • The React framework is a pain. I’ve done my best at the architectural level, but the coupling between React Hooks, business, and UI is too high, preventing full separation. Compared to Signal, it’s too mentally taxing, and I’ll consider migrating to Preact in the future.

  • The persistence solution is terrible, I’m currently directly storing JSON objects, and will switch to Sqlite in the future.

Overall, ATodo is a great tool, worthy of the more than ten days of development time I put into it.

任务规划算法

效果：

  • deadline前一定要达到目标
  • 花的总时间最少

策略：

  • 收集按照任务的依赖关系，收集最小粒度的任务，按照ddl从先排到后
  • 讲执行任务时的条件尽可能最佳，当达到理论最佳时，只能通过减少中断来减少总时间。

此策略与最优策略的差别在于：

  • 需要创造最优条件
  • 在最优条件的前提下，尽可能减少中断造成的额外时间将是唯一的效率提升办法

Definition

Task

What needs to be done:

  • Theoretical Time: The shortest estimated time theoretically needed to complete the task.
  • Conditions: The ratio of the actual time spent to the estimated time, influenced by various factors that need to be clarified, such as:
    • Tool configuration
    • Environmental disturbances
    • Level of fatigue
    • …
  • Actual Time: The real time spent on the task, analyzed through regression for similar tasks.
  • Workload: A unit measurement related to a specific task, which needs to be converted into a standard workload (can be converted into theoretical time).

Task Planning Algorithm

Effectiveness:

  • The goal must be achieved before the deadline.
  • Minimize the total time spent.

Strategy:

  • Collect the smallest granularity tasks according to their dependencies and arrange them from earliest to latest deadlines.
  • Optimize the conditions for task execution as much as possible. When the optimal conditions are achieved, reducing interruptions is the only way to minimize the total time.

The difference between this strategy and the optimal strategy lies in:

  • The need to create optimal conditions.
  • Under optimal conditions, reducing the additional time caused by interruptions will be the only way to improve efficiency.

AToDo - Task Editing Done

Posted on 2023-12-06 In creation

AToDo, 是一款我正在设计与实现的 给我自己用的任务管理工具。

最终的目标

只需要我产生了一个想法，就能通过这个工具的辅助下进行规划。当我讲任务分解，导入（在AToDo规定的框架下）后，我唯一需要做的就是在AToDo领取任务，完成任务。

软件架构

基于tauri的桌面端app。

main线程：rust通过暴露一系列的方法供ui线程进行IPC通信，赋予UI线程访问系统接口的能力。

UI线程：基于系统自带的webview，ui框架为React，单一页面采用一个全局状态，jotai状态管理库用于产生衍生状态以及状态更新。

进展

已完成 基于图与子图数据结构 的任务编辑，任务持久化。

AToDo github

设计任务规划算法。

AToDo is a task management tool that I am currently designing and implementing for my own use.

Ultimate Goal

The goal is for the tool to assist in planning as soon as I have an idea. Once I break down and import tasks (within the framework set by AToDo), all I need to do is to collect and complete tasks in AToDo.

Software Architecture

It’s a desktop app based on Tauri.

Main thread: Rust exposes a series of methods for IPC communication with the UI thread, giving the UI thread access to system interfaces.

UI thread: Based on the system’s webview, the UI framework is React, using a single global state for a single-page application. The jotai state management library is used for generating derived states and state updates.

Progress

I have completed task editing and persistence based on graph and subgraph data structures.
