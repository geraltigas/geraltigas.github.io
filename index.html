任务规划算法

效果：

  • 花的总时间最少

策略：

  • 收集按照任务的依赖关系，收集最小粒度的任务，按照ddl从先排到后
  • 讲执行任务时的条件尽可能最佳，当达到理论最佳时，只能通过减少中断来减少总时间。

  • 需要创造最优条件
  • 在最优条件的前提下，尽可能减少中断造成的额外时间将是唯一的效率提升办法

Definition

Task

What needs to be done:

  • Theoretical Time: The shortest estimated time theoretically needed to complete the task.
  • Conditions: The ratio of the actual time spent to the estimated time, influenced by various factors that need to be clarified, such as:
    • Tool configuration
    • Environmental disturbances
    • Level of fatigue
    • …
  • Actual Time: The real time spent on the task, analyzed through regression for similar tasks.
  • Workload: A unit measurement related to a specific task, which needs to be converted into a standard workload (can be converted into theoretical time).

Segments of the time axis.

Task Planning Algorithm

Effectiveness:

  • The goal must be achieved before the deadline.
  • Minimize the total time spent.

Strategy:

  • Collect the smallest granularity tasks according to their dependencies and arrange them from earliest to latest deadlines.
  • Optimize the conditions for task execution as much as possible. When the optimal conditions are achieved, reducing interruptions is the only way to minimize the total time.

The difference between this strategy and the optimal strategy lies in:

  • The need to create optimal conditions.
  • Under optimal conditions, reducing the additional time caused by interruptions will be the only way to improve efficiency.

React Fiber and Hooks -- from GPT4

Posted on 2023-12-01 In exploration

在 React Fiber 架构下，处理函数组件的过程变得更加高效和灵活。React Fiber 是 React 16 中引入的新的协调算法，它允许 React 暂停和恢复渲染工作，优先级化任务，并且更好地利用多核处理器。以下是在 React Fiber 架构下，处理函数组件（包括其中的 Hooks）的详细过程：

1. 开始渲染阶段

  • Fiber 节点创建：当 React 准备渲染一个函数组件时，它首先为该组件创建一个 Fiber 节点。Fiber 节点是 React 内部用于跟踪组件及其工作状态的数据结构。

2. 函数组件调用

  • 调用函数组件：React 调用函数组件本身，就像调用普通 JavaScript 函数一样。这个调用发生在当前的渲染工作中。

3. Hooks 处理

  • Hooks 调用：在函数组件内部，当遇到 Hooks（如 useState、useEffect）时，React 会处理这些 Hooks。
  • 状态和副作用的追踪：对于每个 Hook，React 在对应的 Fiber 节点上追踪其状态和副作用。例如，useState 会在 Fiber 节点上存储当前的状态值，useEffect 会收集副作用函数及其依赖。

4. 暂停和恢复工作

  • 可中断的工作：Fiber 架构允许 React 暂停正在进行的渲染工作，以优先处理更重要的更新。这是通过 Fiber 节点的结构和工作循环实现的。

5. 完成阶段

  • 渲染输出：函数组件的调用结果（通常是 JSX 或 React 元素）被视为该组件的渲染输出。
  • 子 Fiber 节点：如果渲染输出包含子组件，React 为每个子组件也创建 Fiber 节点，并递归地重复这个过程。

6. 提交阶段

  • 副作用处理：在提交阶段，React 处理所有收集到的副作用（useEffect）。这些副作用在 DOM 更新后执行。
  • 更新 DOM：React 将组件的输出渲染到 DOM 上。这一步是在 Fiber 树的提交阶段完成的。

7. 生命周期和更新

  • 更新：当函数组件的 props 或 state 发生变化时，React 可以重新执行函数组件，更新 Fiber 节点，然后再次进行渲染和提交阶段。

总结

在 React Fiber 架构下，函数组件的处理包括创建 Fiber 节点，调用函数组件，处理 Hooks，暂停和恢复渲染工作，以及最终将组件的输出渲染到 DOM 上。这个架构提高了 React 的性能，特别是在处理大型应用和复杂更新时，同时也使得特性如并发模式（Concurrent Mode）成为可能。

React Principles -- from GPT4

Posted on 2023-12-01 In exploration

阶段

组件初始化

  1. 调用组件函数：
    • 当一个函数式组件首次渲染时，React 调用该组件函数。在这个过程中，执行所有在组件内声明的 Hooks。
  2. 创建 Hooks 链表：
    • 对于每个 Hook 调用（如 useState、useEffect），React 在内部为该组件实例化一个链表节点。每个节点存储该 Hook 的状态和其他相关信息。
    • 这个链表维护了 Hooks 的调用顺序，对于组件的每一次渲染，这个顺序必须保持一致。
  3. 初始化状态和效果：
    • 对于 useState 和 useReducer，React 初始化状态并将其存储在链表节点中。
    • 对于 useEffect，React 记录了副作用函数，但不会立即执行它们。这些副作用函数会在 DOM 更新后的一个微任务中执行。

组件更新（刷新）

  1. 再次调用组件函数：
    • 当组件的 state 或 props 发生变化时，组件会重新渲染。React 再次调用组件函数。
  2. 使用现有 Hooks 链表：
    • React 使用与该组件实例关联的相同 Hooks 链表。对于每个 Hook 调用，React 从链表中检索状态和其他信息。
    • 这确保了每个 Hook 在不同渲染之间保持状态的连续性。
  3. 更新状态和效果：
    • 对于 useState 和 useReducer，React 提供更新函数，允许状态在重新渲染时被更新。
    • 对于 useEffect，React 会比较依赖数组（如果提供的话）来决定是否需要清理上一个效果并运行新的效果。

整体流程

  • 一致性检查：在每次渲染过程中，React 都会检查 Hooks 是否按照相同的顺序和方式被调用。如果发现调用顺序变化，比如由于条件渲染导致的 Hooks 调用顺序不一致，React 会抛出错误。
  • 副作用处理：useEffect 和 useLayoutEffect 在渲染过程中收集副作用函数。对于 useEffect，副作用在渲染完成后的微任务中执行，而 useLayoutEffect 的副作用在 DOM 更新后但浏览器绘制前执行。

总结

React 使用一个精巧的机制来处理函数式组件中的 Hooks。这个机制确保了 Hooks 的状态在组件的不同渲染之间保持一致，同时允许 React 管理组件的生命周期和副作用。通过维护一个固定顺序的 Hooks 链表，并在组件的每次渲染过程中重用这个链表，React 能够高效且准确地处理函数式组件的状态和副作用。

Advanced ToDo List Design - Start(Constructing)

Posted on 2023-11-26 Edited on 2023-11-27 In creation

目的

我适合某一刻时间只做一件事，同时由于存在心流这种现象，做单一的事情长一点时间速度会更加快。频繁的切换任务并不是明智的选择。如果在完成任务时回忆总体计划会打断任务的上下文，并且经常会导致注意力分散。所以需要高效的任务管理机制来自动化任务管理流程。

现在的进展

总体设计

需要一个任务容器

​ 以一定的结构组织任务，实现了用户只需要存入任务，设置任务的属性，即可自动编排任务的执行顺序，ddl，依赖关系以及任务的最佳完成时间。同时任务容器抽象来看只会提供 任务导入，任务取出，以及任务触发检测以及提示 这三个功能。

需要一套任务导入机制

​ 由于用户自身以及其环境这两个来源带来的未经拆分组织的任务往往十分不规整。太大或者太小，无明确ddl，无明确依赖关系，无优先级等等问题。我需要设计一套任务预处理机制，引导用户自行规整任务，然后再导入容器。

任务

定义：需要干的事情

属性（初版）：

  • 结束：客体达到目标状态(需要不断明确，迭代到能够很容易评估的状态，详见下方的任务示例分析)

  • Deadline：结束的最后时间

  • 状态：未开始/挂起/进行中/完成…

  • 挂起触发器：邮件/事件/电话/时间/其他任务/外界环境状态/暂停

  • 依赖关系：与其他任务的关系

  • 子任务: Graph, 要完成该任务需要完成的子任务

Purpose: Conducting a thesis while studying High-Performance Computing (HPC) and Machine Learning Systems (MLSys).

The plan is to work on accelerating LLM (Language Model) inference. A senior recommended a paper titled “Full Stack Optimization of Transformer Inference: a Survey,” which surveyed a key metric for common models called arithmetic intensity:

$$ \frac{FLOPs}{MOPs} $$

Here, MOPs represent the number of memory-related operations (per unit time). The paper indicates that in the case of LLM’s decoder-only transformer during the generation phase, the arithmetic intensity is a constant, approximately equal to 2. This suggests that these operators do not exhibit a significant difference between compute-bound and memory-bound operations.

It’s known that GPUs generally surpass CPUs in both memory access speed and computational speed. If there are many memory-bound operations, they could potentially be shifted to the CPU for processing. Due to the bandwidth limitations of the GPU, memory-bound operations might not differ significantly in speed between the CPU and GPU (although the general belief may vary; GPUs can effectively utilize parallelism in memory access, even though the total bandwidth might be similar, the efficiency could differ substantially).

So far, the benefits of heterogeneity mainly revolve around utilizing the idle CPU. However, this might introduce overhead due to data movement, potentially offsetting the acceleration from CPU processing. Current efforts are moving in this direction.

Regarding parallelism in model inference, the senior mentioned that there might not be many opportunities for operator-level parallelism in transformers. Instead, more parallelism occurs within the operators. This requires further investigation into the operators of common LLM models and attempting intra-operator parallelism.

Further Plans:

  • Utilize model visualization tools to explore common LLM computation graphs and gain a deep understanding of how operators are implemented.
  • Research the underlying implementation of transformer-related operators and study Intel’s vector instruction sets, CUDA programming to support efficient intra-operator parallelism.
